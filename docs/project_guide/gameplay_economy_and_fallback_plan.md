# 玩法增强改造计划：状态工具层 + AI 事件概括兜底

## 1. 目标

本次改造聚焦三个核心问题：

- **策略感不足**：玩家选项多数只是文案差异，缺乏“状态约束 -> 决策代价 -> 结果反馈”闭环。
- **状态逻辑分散**：资金、手机、关系等触发条件散落在不同位置，不利于扩展与调参。
- **事件池空洞风险**：当模组事件量不足或匹配失败时，体验容易掉到兜底文案，剧情断层明显。

对应改造目标：

1. 引入“**状态工具层**”机制，让资金/理智/学业/卫生/名誉/关系阶段都走统一判定与结算。
2. 引入“**AI 事件概括兜底**”机制，在事件池不足时生成结构化事件草案，由系统托管执行。

---

## 2. 设计原则

- **系统先行，AI 表现**：规则判定和状态结算由系统负责；AI 只负责文本化表达。
- **可解释**：玩家必须看到“为什么不可选/为什么属性变化”。
- **可控兜底**：AI 兜底只生成“概括骨架”，不直接改状态、不直接跳过规则。
- **兼容旧内容**：现有 CSV/骨架不改也能跑，新字段按“可选”处理。
- **一处扩展，多处复用**：手机消息触发、关键选项门槛、关系里程碑都优先复用状态工具层。

---

## 3. 功能 A：状态工具层（P1）

## 3.1 目标接口（统一动作）

- `state_check`：检查条件是否满足（可返回失败原因）
- `state_apply`：执行状态变更（支持批量 delta / 标记）
- `state_explain`：将判定结果转成 UI 可读短句（禁用原因、消耗说明）

说明：可以先在后端内部实现为函数，不强制第一版就做独立 API。

## 3.2 数据结构（扩展关键选项）

在关键事件选项（`key_options`）上新增可选字段（通用状态版）：

- `require`: 条件对象（支持 money/san/gpa/hygiene/reputation/关系阶段）
- `cost`: 消耗对象（支持多属性）
- `effects`: 保持现有逻辑（变化可继续走已有 effects）
- `fail_hint`: 条件不满足时的提示文案（可选）

示例：

```json
{
  "label": "答应一起出去吃夜宵",
  "attitude": "support",
  "effects": [{"type":"relationship_delta","target":"lin_sa","trust":3}],
  "require": {
    "money_gte": 60,
    "hygiene_gte": 30,
    "relationship_stage_gte": {"target":"lin_sa","stage":"familiar"}
  },
  "cost": {"money": 60, "san": 2},
  "fail_hint": "余额不足，没法请客。"
}
```

## 3.3 执行规则

1. 回合渲染选项前检查 `require`。  
2. 不满足条件时：  
   - 前端显示“禁用态”（灰置）  
   - 保留可见文案，附原因（`fail_hint` 或 `state_explain`）  
3. 玩家提交被禁用选项时，后端二次校验并拒绝，返回明确错误。  
4. 选项结算时先执行 `cost`（可多属性），再执行原有 `effects`。  

## 3.4 前端表现

- 选项右侧可显示轻量代价标签：`-¥60 / -SAN2`
- 禁用态选项显示：`需要资金≥60、卫生≥30`
- 结算后在文本中增加系统反馈短句：`（已消费 60 元，理智 -2）`

## 3.5 兼容策略

- 无 `require/cost` 的旧选项行为不变。
- 若字段非法（如负数、非数字），校验阶段给 warning，运行时按“无门槛”降级。

## 3.6 手机系统与状态工具的统一

- `phone_enqueue_message` 的触发条件也走 `state_check`（例如关系阶段、最近事件标签、冷却回合数）。
- 这样手机消息与关键选项共享同一判定来源，避免“发消息很随机”。
- 调试面板可统一展示：`state_rule_hit` / `state_rule_miss`。

---

## 4. 功能 B：AI 事件概括兜底（P1.5）

## 4.1 触发时机

仅在以下情况触发兜底：

- 当日关键事件候选池为空，或全部被条件过滤掉
- 日常事件抽取失败且可用候选不足

正常情况下优先走配置事件，不抢主流程。

## 4.2 兜底输出契约（新 JSON 结构）

新增一个“事件概括”响应契约（不是完整剧情）：

- `title`: 事件标题
- `scene`: 场景名
- `hook`: 冲突引子（1 句）
- `choices`: 3 条态度选项（support / neutral / avoid）
- `effects_hint`: 每个选项的建议影响（可为空）

示例：

```json
{
  "title": "预算分配分歧",
  "scene": "宿舍书桌区",
  "hook": "室友提议合买物品，但预算意见不一致。",
  "choices": [
    {"attitude":"support","copy":"支持集体购买，提议你多出一点"},
    {"attitude":"neutral","copy":"建议先列清单再决定"},
    {"attitude":"avoid","copy":"先回避，表示下次再谈"}
  ],
  "effects_hint": [
    {"attitude":"support","money_delta":-40,"dorm_mood_delta":2},
    {"attitude":"neutral","dorm_mood_delta":1},
    {"attitude":"avoid","dorm_mood_delta":-1}
  ]
}
```

## 4.3 系统接管

- 后端将上述概括转为临时 `ScriptedEvent`（内存态）并进入常规回合结算。
- 状态变化仍走系统状态机；AI 不直接写状态。
- 记录 `source=fallback_summary` 到调试字段，便于统计。

## 5. 实施顺序

## 阶段 1（优先）

- 状态工具核心函数：`state_check/state_apply/state_explain`
- 关键选项接入 `require/cost/fail_hint`
- 首批接入属性：`money`（基线）
- 前端选项禁用态 + 代价标签

当前进度（2026-03-24）：
- 已完成：关键事件选项 `require/cost/fail_hint` 基础链路（money 首批）
- 已完成：回包 `next_options_meta`（禁用态/原因/代价标签）
- 已完成：前端选项禁用显示与原因提示
- 已完成：后端二次校验（条件不满足时拒绝该关键选项）
- 待补：编辑器可视化配置 `require/cost`（仍可手改 skeleton/json）

## 阶段 2

- 扩展属性：`san/gpa/hygiene/reputation` 与 `relationship_stage`
- 手机消息触发条件改走状态工具层
- 增加 AI 概括兜底生成接口（可复用当前 expression 模式模型调用）
- 概括 -> 临时事件映射器
- 调试面板显示事件来源（configured / fallback_summary）

## 阶段 3

- 编辑器支持可视化编辑门槛与成本字段
- 统计页增加“门槛触发次数 / 兜底触发次数”

---

## 6. 验收标准

- 玩家在低属性时能明显感知“某些选项不可选”，且原因清楚。
- 有消耗选项结算后有即时反馈（资金/属性变化可见）。
- 事件不足时不会掉入空白兜底文案，而是稳定进入“概括事件”流程。
- 旧模组不修改也可运行。

---

## 7. 风险与防护

- **风险：** 条件过严导致可选项过少。  
  **防护：** 至少保留 1 个“无门槛基础选项”。

- **风险：** AI 概括质量波动。  
  **防护：** 契约严格 + 后端二次校验 + 不合法则回退到安全默认事件。

- **风险：** 玩家误解“禁用=Bug”。  
  **防护：** UI 明确写出禁用原因和达成条件。

---

## 8. 下一步建议

优先先做“状态工具 + money 首批接入”这一条，改动小、收益高；跑一轮体验后，再扩到其余属性并接“AI 事件概括兜底”。
