# 叙事记忆系统设计（v1）

本文档面向 `AI_Dialogue` 的核心玩法目标：  
让 AI 长期扮演 DM，模拟主角四学年的大学宿舍生活，在接近 100 次 API 调用中保持角色一致性、关系连续性与世界细节可信度。

## 1. 需求摘要

目标体验：

- 主角与舍友关系可持续演化（普通同学 -> 挚友/恋爱/仇敌）
- 事件推进中，数值与关系变化可追溯、可解释
- 世界观细节丰富（学校、教师、配角、语录）但不“AI 味”
- 长流程中不失忆、不跳戏、不反复重置关系

规模约束：

- 四个学年
- 每学年 3-4 个事件
- 每事件 3-6 轮
- 总调用量约 80-100 次

结论：需要“分层记忆 + 可控状态机 + 按需注入 Prompt”，而不是单纯堆上下文。

## 2. 设计原则

1. 状态优先：所有关系/属性变化都必须落到结构化状态，不依赖 LLM 自己记住。  
2. 最小上下文：每回合只注入“当前决策必需”的信息，避免全量设定重复塞入。  
3. 可解释结算：变化由 `effects` 驱动，保证可审计、可回放。  
4. 多层摘要：短期原文 + 中期事件摘要 + 长期里程碑并存。  
5. 叙事连续性优先于文采：先确保人设稳定和因果闭环，再追求语言华丽。

## 3. 记忆分层模型

### 3.1 短期记忆（Turn Memory）

用途：维持当前回合衔接与台词连贯。  
内容：

- 最近 1-2 回合 `dialogue_sequence`
- 玩家上一选择与即时结果
- 当前事件目标（opening/pressure/turning/settlement）

注入策略：每回合必注入，严格限长（字符预算优先级最高）。

### 3.2 中期记忆（Event Memory）

用途：保证单事件 3-6 轮内不跑偏。  
内容：

- 本事件已发生的关键分歧
- 已触发的冲突点、已兑现的关系变化
- 事件未完成目标（本轮必须推进什么）

写入时机：每回合结算后更新。  
注入策略：每回合注入摘要版，不注入完整历史文本。

### 3.3 长期记忆（Arc Memory）

用途：跨学年持续人设与关系轨迹。  
内容：

- 关系里程碑（和解、决裂、暧昧、公开冲突等）
- 角色标签漂移（“嘴硬但愿意帮主角”）
- 世界事实变更（教师态度、社团地位、宿舍规则）

写入时机：事件结束、章节切换、学年结束。  
注入策略：按角色相关性检索 Top-K，禁止全量注入。

## 4. 关系状态模型（建议字段）

每个主要角色建议维护：

- `affinity`：0-100（已有）
- `trust`：0-100
- `tension`：0-100
- `intimacy`：0-100（用于亲密关系线）
- `relationship_stage`：`陌生/熟悉/朋友/暧昧/恋人/敌对` 等
- `recent_flags`：最近 3-5 个关系标签（如“被背刺”“被维护”）
- `last_milestone_at`：最近里程碑时间点（turn/chapter）

说明：`affinity` 不能单独表达复杂关系，`trust/tension/intimacy` 可显著降低“好感高但行为突变”的违和。

## 5. 回合处理流水线（v1）

1. 读取当前事件与玩家输入  
2. 组装 Prompt（短期 + 中期 + 相关长期 + 必要世界设定）  
3. LLM 输出 JSON（`dialogue_sequence / next_options / effects / is_end`）  
4. 后端解析并执行 `effects`  
5. 更新关系状态与记忆层  
6. 生成下一回合可用摘要（供下一轮注入）

关键点：第 4-5 步必须在后端确定，不交给 LLM 自行“脑补结算”。

## 6. Prompt 注入预算建议

建议按预算分块（与现有 `static/repeated/dynamic/trimmable` 对齐）：

- `dynamic`：当前事件目标、玩家动作、上一轮结果（必注入）
- `repeated`：在场角色速览、关系速览（短摘要）
- `trimmable`：RAG 语录、世界细节扩展、微信历史
- `static`：系统规则、JSON 合约、作者注释

推荐默认策略（提速友好）：

- 普通回合：`compact`
- 关键事件/收束回合：`full`
- 教学区等特定场景才注入对应世界配角块（按需加载）

## 7. 与当前代码的落点映射

可复用基础：

- `GameEngine._build_user_prompt_sections`：已具备分块拼接框架
- `NarrativeStateManager`：可扩展中期/长期摘要输出
- `effects` 结算链路：已可作为关系状态更新入口
- `PromptManager.skills`：可继续演进为“触发式记忆/设定插件”

优先改造点：

- 将长期记忆写入从“松散文本”收口到结构化记录
- 建立“关系状态更新器”（统一处理 trust/tension/intimacy）
- 建立“事件总结器”（事件结束时压缩成长期里程碑）

## 8. Skill / MCP 在本项目的定位

### Skill（建议立即强化）

建议作为“本地可控插件层”继续使用，增加：

- `trigger_condition`
- `priority`
- `token_budget`
- `fallback_behavior`

目标：减少无关信息注入，提高稳定性与速度。

### MCP（暂不必强依赖）

当前体量下不建议先上完整 MCP 协议。  
更推荐先把本地工具与数据检索边界整理清楚，后续若接入大量外部服务（云知识库、检索网关、社区系统）再迁移到 MCP。

## 9. 分阶段实施建议

### Phase 1（高优先级）

- 建立关系状态扩展字段（trust/tension/intimacy/stage）
- 统一通过 `effects` 更新关系状态
- 事件结束写入里程碑摘要（长期记忆）

### Phase 2（高优先级）

- 增加“记忆注入预算器”（按回合类型切换 compact/full）
- 将世界细节改为“场景触发注入”，避免全局常驻

### Phase 3（中优先级）

- Skill 元数据化（触发条件/预算/优先级）
- 建立记忆质量评估指标（连续性、复读率、人设漂移率）

## 10. 验收指标（建议）

- 连续 30 回合内角色关系无明显反向跳变
- 重复选项率明显下降（可按事件统计）
- 平均回合耗时下降，且 JSON 结构错误率不上升
- 玩家主观评价：人物“像同一个人”而不是每回合重开

## 11. 当前实现进度（2026-03-21）

已落地（Phase 1 的核心骨架）：

- `NarrativeStateManager` 新增 `relationship_state` 结构，包含 `trust/tension/intimacy/relationship_stage/recent_flags/last_milestone_at`
- 每回合基于 `effects + action + dialogue` 更新关系状态，并生成关系标签
- 关系阶段变化会写入 `long_term_milestones`
- 事件结束会写入一条关系快照里程碑

- 里程碑已复用现有 `MemoryManager + VectorStore` 写入 `narrative_milestone` 记忆类型（带 `save_id`）
- 回合构建 prompt 时会按 `save_id` 从向量库检索相关里程碑并以 `trimmable` 方式注入
- 增加关系阶段防抖/迟滞规则，减少一两轮噪音导致的阶段抖动

- 新增窗口机制单测（里程碑触发窗口、关系标签匹配）
- 已实现模板事件实例化（`[TARGET]` 占位符自动绑定当前关系最差在场角色）
- 模板事件会生成目标角色专用事件 ID（避免预取缓存串角色）
- `02_通用随机池.csv` 已补 6 条关系模板事件样本（冲突/冷战/和解/暧昧/邀约/公开站队）

## 12. 关系变化不明显的解决方案（事件联动）

### 12.1 问题本质

当前系统已经能更新关系状态，但“状态 -> 剧情反馈”链路还偏弱。  
表现为：

- 玩家看到好感/关系阶段变化，但下一事件仍像普通随机事件
- 缺少“关系转折后必然发生的戏剧节点”（如吵架、和解、约会、冷战）

这会导致体验上“数值在变，剧情没变”。

### 12.2 设计目标

让关系变化直接影响事件分发：

1. 关系恶化后更容易触发冲突类事件（吵架、冷处理、误会升级）  
2. 关系升温后更容易触发亲密类事件（单独相处、约会、互相维护）  
3. 阶段跃迁时可强制插入“过渡事件”（从朋友到暧昧，不应无过渡）

### 12.3 可复用现有能力（避免“屎上加屎”）

已有基础可以直接利用：

- `EventDirector._event_weight(event, narrative_state)` 已支持按叙事状态加权
- 事件定义已有 `narrative_tags / trigger_conditions / event_weight / cooldown`
- `narrative_state` 现已包含 `relationship_state` 与 `long_term_milestones`

因此建议做“增量改造”，而不是新建一套独立调度器。

### 12.4 推荐改造方案（低侵入）

#### A. 扩展标签体系（先文档约定，再补事件）

建议在事件 `narrative_tags` 中增加关系驱动标签：

- `关系-升温`
- `关系-恶化`
- `关系-和解`
- `关系-暧昧推进`
- `关系-冷战`
- `关系-公开冲突`

这一步只改数据，不改主流程。

#### B. 在 `_event_weight` 增加关系权重规则

参考现有 `room_tension/player_arc/active_threads` 加权逻辑，增加：

- 若某在场角色 `relationship_stage=紧张/敌对`：提高 `关系-恶化/公开冲突/冷战` 类事件权重
- 若某在场角色 `relationship_stage=朋友/暧昧/恋爱倾向`：提高 `关系-升温/暧昧推进/和解` 类事件权重
- 若最近里程碑出现“阶段变化”：对应过渡类事件临时加权（仅 1-2 次）

#### C. 增加“阶段变化后保底事件窗口”

当发生关系阶段变化时，记录一个短期窗口（如未来 2-3 个事件）：

- 在窗口内，至少命中 1 个与该变化匹配的事件标签
- 若普通加权未命中，则在候选池中执行一次“关系标签优先”兜底

这样可以保证“变差/变好”一定有戏剧反馈。

#### D. 控制密度，避免过拟合

- 每学年关系强反馈事件占比建议 25%-40%
- 剩余仍保留世界观/学业/日常事件，防止系统变成“纯恋爱/纯撕逼模拟器”

### 12.5 事件数据层改造建议

建议在 CSV 或事件对象新增可选字段（可逐步补）：

- `relationship_target`: 该事件主要作用于哪位角色（如 林飒）
- `relationship_phase_gate`: 触发所需阶段（如 `紧张以上`、`朋友以上`）
- `relationship_effect_type`: 主要作用方向（升温/恶化/和解）
- `relationship_priority`: 关系反馈优先级（0-3）

若短期不改 schema，也可先复用 `narrative_tags + trigger_conditions` 实现。

### 12.6 实施顺序（建议）

1. 先做 `_event_weight` 的关系规则（代码改动小，收益立竿见影）  
2. 再补 8-12 个“关系反馈事件”样本（每章 2-3 个）  
3. 最后加“阶段变化后保底事件窗口”机制  

### 12.7 验收指标（关系反馈维度）

- 关系阶段变化后，3 个事件内至少出现 1 个相关反馈事件
- 玩家主观感受：能明确感到“刚吵完架，后续确实更别扭/更冲突”
- 玩家主观感受：升温后能看到明确正反馈场景（非仅文本里一句“关系变好”）
