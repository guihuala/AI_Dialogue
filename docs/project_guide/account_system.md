# 账户系统说明

本文档说明 `AI_Dialogue` 当前账户系统的设计目标、落地方案、实现进度与后续缺口。

## 概述

- 后端已支持：
  - `POST /api/account/register`
  - `POST /api/account/login`
  - `GET /api/account/me`
  - `POST /api/account/change_password`
  - `POST /api/account/bind_current_visitor`
  - `GET /api/account/visitor_binding_preview`
  - `GET /api/account/sessions`
  - `POST /api/account/logout_others`
  - `POST /api/account/revoke_session/{session_id}`
  - `POST /api/account/logout`
  - `GET /api/workshop/mine`（需登录）
  - `GET /api/admin/users`（管理员）
  - `GET /api/admin/users/stats`（管理员）
- 已支持密码哈希存储，不保存明文密码
- 已支持服务端会话与过期控制
- 已支持请求身份优先级：
  - 先识别 `account_token`
  - 再回退 `visitor_id`
- 已保留匿名访客模式
- 注册时已支持绑定当前访客数据
- 已支持绑定前预览当前访客数据规模与潜在重名冲突
- 已支持绑定时选择冲突处理策略：
  - `keep_account`（保留账户已有内容）
  - `overwrite_with_visitor`（用访客内容覆盖冲突）
- 已支持返回迁移报告（新增 / 覆盖 / 冲突跳过）
- 已开始将“公开归属类动作”收口为登录后才可执行：
  - 工坊发布 / 更新 / 派生发布现要求正式账户登录
- 前端提供独立账户入口
- 前端支持：
  - 查看当前身份状态
  - 注册
  - 登录
  - 登出
  - 修改密码
  - 绑定当前访客数据
  - 查看登录会话
  - 退出其他设备
  - 移除指定旧会话
  - 查看最近活动
  - 直接展示当前身份的能力范围（访客可用 / 登录后可用）
  - 在发布模组时对“需先登录账户”给出明确提示与跳转引导
  - 在工坊页对“我的作品”等登录后能力给出显式限制提示
  - 顶部显示当前是访客还是已登录

### 完成

- 访客转正：
  - 当前已支持“注册时绑定当前访客数据”和“登录后单独绑定当前访客数据”
  - 当前已支持绑定前预览与冲突提示
  - 当前已支持冲突策略，但还没有逐文件交互式确认向导

- 账户中心：
  - 当前已支持账户资料、密码修改、会话管理、最近活动
  - 但还没有更细的会话设备信息、登录地点、会话命名等增强字段

- 后台用户管理：
  - 当前已支持管理员用户列表检索、分页和基础统计
  - 更深的账户治理能力（封禁、导出、批量审计）

- 更细的访客迁移确认与逐项冲突处理
  
- 系统的“哪些动作必须登录后才能做”的权限梳理
  - 当前已开始显式展示能力范围，并已收口工坊发布类动作
  - 工坊“我的作品”已改为登录专用接口 `/api/workshop/mine`
  - 但仍可继续细化更多接口级权限边界

### 进度判断

- 第一阶段基础账户系统：约完成 `80%-85%`
- 账户系统长期目标：约完成 `60%-70%`

## 设计目标

账户系统的目标尽量克制，优先解决数据归属问题：

- 让玩家拥有稳定、可重复登录的账号身份
- 让现有存档、模组、工坊作品继续按用户隔离
- 保留匿名试玩能力，不强迫一开始就注册
- 支持将匿名访客数据升级绑定到正式账号
- 尽量继续沿用现有文件式存储，不立刻引入数据库

非目标：

- 暂不做邮件验证
- 暂不做找回密码邮件
- 暂不做第三方登录
- 暂不做多人协作编辑

## 方案

采用两层身份模型：

### 4.1 访客身份

- 继续保留当前 `visitor_id`
- 用于未登录状态下的试玩
- 仍然可以生成独立的用户目录

### 4.2 正式账户身份

- 新增 `account_id`
- 通过用户名 + 密码进行注册和登录
- 登录成功后由后端签发 `account_token`
- 后续请求优先使用 `account_token` 识别身份，而不是只看 `visitor_id`

简化理解：

- `visitor_id`：临时试玩身份
- `account_id`：正式长期身份
